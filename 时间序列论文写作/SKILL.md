# 时间序列论文写作：段落级结构蓝图

为撰写投稿顶级 ML 会议（NeurIPS、ICML、ICLR、AAAI、KDD、IJCAI）的**时间序列预测论文**提供精细化结构指导。本技能提供章节页数分配、段落级蓝图、标准数据集/基线/指标速查表，以及时间序列方向特有的审稿痛点清单。

## 9 页 ML 会议论文蓝图

### 总览：页数分配（以 8–9 页正文 + 不限页数附录为例）

| 章节 | 页数 | 目的 |
|---------|-------|---------|
| 摘要 | ~0.2 | 150–200 字，五句话结构（见 lab-style-guide.md） |
| S1 引言 | 1–1.5 | 问题 → 现有方法局限 → 方案 → 贡献 |
| S2 相关工作 | 0.5–1 | 按方法范式分组，而非逐篇罗列 |
| S3 方法 | 2–2.5 | 问题定义 + 框架总览 + 逐模块设计 |
| S4 实验 | 3–3.5 | 设置 + 主实验 + 消融 + 效率/超参敏感性 + 可视化 |
| S5 结论 | 0.3 | 三句话总结 |
| **合计** | **~8–9** | NeurIPS/ICML/ICLR 主会一般 9 页正文（不含参考文献），AAAI/IJCAI 通常 7–8 页。附录页数不限，用于补充实验、证明与超参细节。投稿前务必核对当年 CFP。|

### 摘要（150–200 字，五句话）

```text
第 1 句：领域定位（时间序列预测在 XX 决策中的关键作用）
第 2 句：现有方法的局限（统计方法/RNN/Transformer/LLM-based 各自的不足）
第 3 句：核心方案（提出 ModelName，核心机制是什么）
第 4 句：模块拆解（1-3 个子模块各自的作用）
第 5 句：实验结论（N 个真实数据集，K 个应用领域，达到 SOTA）
```

详细句式模板见机器学习论文写作技能的 `references/lab-style-guide.md`（本技能不重复）。

### S1 引言（1–1.5 页）

**段落结构**：

1. **问题陈述**（~0.3 页）—— 时间序列预测的应用场景与重要性（能源、交通、金融、气象等），每个应用配 2–3 篇引用体现覆盖广度。
2. **方法演进与差距**（~0.5 页）—— 统计方法（ARIMA）→ 深度模型（RNN/TCN）→ Transformer 类（Informer/Autoformer/FEDformer）→ 线性/MLP 类（DLinear/TiDE）→ LLM-based。明确指出当前范式的具体差距（长程依赖、分布漂移、计算开销、可解释性等）。
3. **核心方案**（1 段）—— 一句话论题，说明核心机制与解决的具体差距对应关系。
4. **贡献列表**（~0.3 页）—— 3–4 项编号贡献，倒数第二项写方法贡献，最后一项固定写实验结论（"在 N 个真实数据集上达到 SOTA"）。

**来源模式**：差距枚举（G1–Gn）在方法章节逐一对应解决方案，与读者建立清晰契约——这一模式与系统论文的"差距分析"范式同构，可直接套用。

### S2 相关工作（0.5–1 页）

- 按**技术范式**分组：统计方法 / RNN-based / CNN-based / Transformer-based / MLP-Linear-based / LLM-based / 扩散模型-based，而非逐篇罗列。
- 每组：代表性工作、核心思路、局限。
- 若本文与某一组方法关系紧密（如同属 Transformer-based），用对比表格列出关键设计维度差异（如是否 channel-independent、是否显式建模周期性、计算复杂度）。

### S3 方法（2–2.5 页）

**段落结构**：

1. **问题定义**（~0.3 页）—— 形式化输入回看窗口（look-back window，长度 $L$）、预测窗口（horizon，长度 $H$）、多变量/单变量设定，给出统一符号表。
2. **框架总览**（~0.3 页）—— 先放架构图，一段话介绍各模块与数据流（"Figure 1 illustrates the overall framework of ModelName, which consists of N main components..."）。
3. **逐模块设计**（~1.5 页）—— 每个子模块：动机、具体设计、与已有方案的差异、关键公式。
4. **复杂度/可解释性讨论**（可选，~0.2 页）—— 时间序列论文审稿人常关心计算复杂度（与 $L$、$H$、变量数 $C$ 的关系），主动给出有助于降低"效率担忧"。

### S4 实验（3–3.5 页）

**段落结构**：

1. **实验设置**（~0.5 页）—— 数据集（见下方标准数据集表）、基线（见下方标准基线表）、评估协议（look-back/horizon 组合、滚动预测 or 单次预测、归一化方式）、指标（MSE/MAE，分类任务用 Accuracy/F1）。
2. **主实验结果**（~1–1.5 页）—— 多数据集 × 多 horizon 的大表格，最优结果加粗、次优下划线；正文用 1–2 段总结整体趋势，不逐格读数。
3. **消融实验**（~0.5–1 页）—— 见下方"消融实验设计清单"，逐一移除/替换模块验证贡献。
4. **效率与超参敏感性**（~0.3–0.5 页）—— 训练/推理时间、参数量对比；关键超参（如 look-back 长度、patch size、隐藏维度）的敏感性曲线。
5. **可视化分析**（~0.3 页）—— 预测曲线对比图、注意力/权重可视化，增强可解释性说服力。

**关键规则（与系统论文评估章节同构）**：每个核心实验结论陈述**三次**——节首假设、节尾结论、图表标题中的证据句。

### S5 结论（0.3 页）

三句话：方案是什么 → 核心机制 → 实验结论。详细句式见 lab-style-guide.md。

## 标准数据集速查表

| 领域 | 数据集 | 规模/特点 |
|------|--------|-----------|
| 能源/电力 | ETT (ETTh1/h2/m1/m2)、Electricity | 多变量，强周期性，长程预测常用基准 |
| 交通 | Traffic、PeMS | 高维多变量（800+ 变量），适合测试可扩展性 |
| 气象 | Weather | 21 个气象指标，非平稳性较强 |
| 金融 | Exchange Rate | 变量间相关性弱，长程预测难度高 |
| 疫情 | ILI | 短序列、低频，适合小样本场景 |
| 工业 | Solar-Energy | 强稀疏性（夜间为零） |
| 短期预测 | M4 | 多频率（年/季/月/周/日/小时），常用于短期单变量预测评测 |

## 标准基线速查表

| 范式 | 代表方法 | 备注 |
|------|----------|------|
| 统计方法 | ARIMA、Prophet | 仅作传统方法对照，非主要对比对象 |
| RNN/TCN | LSTM、DeepAR、TCN | 长程预测中通常表现较弱，但需保留以体现演进脉络 |
| Transformer-based | Informer、Autoformer、FEDformer、Crossformer、PatchTST、iTransformer | **PatchTST、iTransformer 是近年强基线，缺席易被审稿人质疑** |
| MLP/Linear-based | DLinear、TiDE、TSMixer、TimeMixer | **DLinear 是著名的"简单方法打败复杂 Transformer"基线，几乎是必报项** |
| LLM-based | Time-LLM、TimeGPT、Chronos | 若论文方向涉及 LLM，需纳入对比 |

> ⚠️ **审稿高频雷区**：缺少 DLinear、PatchTST、iTransformer 中的任意一个常会被审稿人直接点名要求补充。投稿前核对最新一年顶会论文的基线列表，基线集合每年水涨船高。

## 消融实验设计清单

- [ ] 逐一移除每个核心模块，验证各自贡献（而非只做整体 vs baseline）
- [ ] look-back window 长度敏感性（如 $L \in \{96, 192, 336, 720\}$）
- [ ] 预测 horizon 敏感性（短程 vs 长程，验证方法是否仅在某一区间有效）
- [ ] channel-independent vs channel-dependent 设计的对比（若涉及多变量建模）
- [ ] 计算效率对比（参数量、训练/推理时间，尤其是声称"高效"时必须有此项）
- [ ] 替换核心组件为简单基线（如把提出的注意力机制换成普通自注意力），证明设计的必要性而非仅靠堆参数量

## 时间序列论文常见审稿痛点

| 痛点 | 如何预防 |
|------|----------|
| "为什么没有和 DLinear/PatchTST/iTransformer 比？" | 主实验表格必须覆盖当前年度的强基线，缺席要在限制中说明原因 |
| "只在标准基准上测试，缺乏真实部署场景验证" | 补充至少一个非标准/实际业务数据集，或在限制中坦诚讨论 |
| "性能提升来自更大参数量而非设计创新" | 给出参数量/计算量对比表，证明同等或更小开销下的提升 |
| "消融实验不充分，无法判断每个模块的真实贡献" | 逐模块消融 + 替换式消融（而非只删不换） |
| "长程预测（如 horizon=720）性能骤降未讨论" | 在敏感性分析中专门讨论长程退化现象及原因 |
| "评估协议与基线论文不一致（look-back 长度、归一化方式不同）导致结果不可比" | 明确写出与基线完全一致的评估协议，或注明协议差异并解释 |
| "图表标题/表格缺乏统计显著性说明" | 多次随机种子实验报告均值±标准差，或做显著性检验 |

## 写作理念

### 数据集×基线全覆盖优先于方法新颖性堆叠
时间序列预测领域审稿人对"实验不全"的容忍度远低于对"方法不够新"——先保证标准数据集和基线的完整覆盖，再追求方法创新的深度。

### 效率是默认审查项，不是加分项
近年审稿人普遍要求时间序列论文给出参数量/训练时间对比，尤其当方法基于 Transformer 或 LLM 时，"效率讨论缺失"几乎等同于"实验不完整"。

### 第一页图（与系统论文写作同构）
在摘要后第一页放整体框架图，帮助审稿人快速建立方法的直觉印象。

## 学术诚信要求

### 引用规范
- **绝不凭记忆生成引用。** 使用机器学习论文写作的引用核实工作流（Semantic Scholar / DBLP / CrossRef API）。
- 将未经核实的引用标记为 `[CITATION NEEDED]`。

### 禁止捏造
- 不捏造实验结果、数据集统计量或基线性能数字。基线数字应引用原论文或权威复现库（如 Time-Series-Library），并注明来源。
- 不生成虚假的会议规则、论文元数据或最佳论文声称。
- 不从参考论文复制段落级文本。本蓝图提供**结构指导**，而非复制粘贴模板。

### LLM 披露
- 部分会议要求披露在写作或构思中大量使用 LLM。请查阅当年 CFP 中的 AI 政策。

### 时效性
- 标准基线集合每年变化（新强基线不断出现），数据集惯例也会调整。本技能中的数据集/基线表为常见基准集合，**投稿前务必核对近两年同会议论文的基线选择**。

## 工作流：构建新时间序列论文的结构

```text
步骤 1：阅读本 SKILL.md 了解页数分配概览与领域知识速查表
步骤 2：按蓝图逐节起草（S1 引言 → S2 相关工作 → S3 方法 → S4 实验 → S5 结论）
步骤 3：对照标准数据集/基线速查表，检查实验覆盖是否充分
步骤 4：对照消融实验设计清单与审稿痛点表，自查实验设计
步骤 5：使用 机器学习论文写作 的 lab-style-guide.md 进行段落级句式润色
步骤 6：使用 机器学习论文写作 进行引用核实和 LaTeX 格式化
```

### 快速检查清单
